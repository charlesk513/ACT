/******************************************************************************
** Task: Running the slow program
**
** 1. What is the complexity of findSimilarity?
**    Answer in terms of N, the total number of 5-grams in the input files.
**    Assume that the number of 5-grams that occur in more than one file
**    is a small constant - that is, there is not much plagiarised text.
******************************************************************************/

The complexity of findSimilarity is O(N^2).

Here is why:
- Let D = number of documents and K = number of 5-grams per document.
- The algorithm has 4 nested loops: for each of the D documents (d1),
  for each of the D documents (d2), for each of K n-grams in d1, and
  for each of K n-grams in d2.
- That gives D * D * K * K = D^2 * K^2 operations.
- Since N = D * K (total n-grams), we have D^2 * K^2 = (D*K)^2 = N^2.
- Therefore the complexity is O(N^2).

small  directory (N ≈ 20,000):   approximately 2–3 seconds

/******************************************************************************
** Task: Using binary search trees
**
** 4. Which of the trees usually become unbalanced?
******************************************************************************/

The 'index' BST (BST<Ngram, ArrayList<Path>>) and the 'similarity' BST
(BST<PathPair, Integer>) tend to become unbalanced.

The 'files' BST is usually fine because file paths are inserted in sorted
alphabetical order from the file system, which creates a reasonably balanced tree.

The 'index' BST becomes unbalanced because 5-grams are inserted in the order
they appear in the files, and if the documents contain a lot of repeated or
alphabetically ordered words (as in the badforbst directory), insertions follow
a nearly sorted order that degenerates the BST into a linked list.

The 'similarity' BST can also become unbalanced for similar reasons - PathPairs
are processed in a pattern that may produce nearly sorted insertions.


/******************************************************************************
** 5 (optional). Is there a simple way to stop these trees becoming unbalanced?
******************************************************************************/

Yes. One simple approach is to use a balanced BST such as a scapegoat tree,
AVL tree, or red-black tree instead of a plain BST. These data structures
automatically rebalance themselves when nodes are inserted.

Another simple approach is to shuffle the input before inserting into the BST,
since random insertion order produces a much more balanced tree on average.
However, this is not always easy to do and does not give a worst-case guarantee.

The cleanest fix for this lab is to replace BST with ScapegoatTree
(as done in the next task), which provides a worst-case height guarantee.


/******************************************************************************
** Task: Using scapegoat trees
**
** 6. Now what is the total complexity of buildIndex and findSimilarity?
**    Again, assume a total of N 5-grams, and a constant number of 5-grams
**    that occur in more than one file.
******************************************************************************/

buildIndex complexity: O(N log N)
  - We iterate over all N 5-grams across all files.
  - Each 5-gram is inserted into (or looked up in) the scapegoat tree,
    which takes O(log N) time.
  - Total: O(N log N).

findSimilarity complexity: O(N log N)
  - We iterate over all keys in the index. There are at most N unique 5-grams.
  - For each 5-gram, we look at all pairs of files that share it.
  - Since we assumed the number of shared 5-grams is a small constant, the
    number of file pairs per n-gram is also bounded by a constant.
  - For each pair we do one insert/update in the similarity BST: O(log N).
  - Total: O(N log N).

Combined complexity: O(N log N) for the whole program (excluding readPaths).


/******************************************************************************
** 7 (optional). What if the total similarity score is an arbitrary number S,
**               rather than a small constant?
******************************************************************************/

If the total similarity score across all pairs is S (meaning there are S
pairs of matching 5-grams), then:

- findSimilarity would need O(S log N) time because we process S matching pairs
  and each update to the similarity BST costs O(log N).

- buildIndex still takes O(N log N).

So the total complexity becomes O((N + S) log N).

In the worst case where all documents are identical, S could be O(N^2),
giving O(N^2 log N) total — which is worse than the original O(N^2)!
This shows that the index-based approach is only faster when there is
not too much plagiarism (S is small compared to N^2).

** B. Are there any known bugs / limitations?
******************************************************************************/

No known bugs. The program correctly:
- Builds the n-gram index in buildIndex.
- Uses the index to efficiently compute similarity in findSimilarity.
- Implements the scapegoat tree (put, inorder, balanceNodes) correctly.

One limitation: on very large document sets, the recursive rebuild()
in ScapegoatTree could cause a stack overflow if called on a subtree
with many thousands of nodes. An iterative implementation of inorder
would avoid this but was not needed for the provided document sets.

** D. Describe any serious problems you encountered.

The main challenge was correctly implementing the scapegoat tree's put method.
We had to make sure that the size and height fields were updated correctly after
every insertion, and that the balance check used the right formula
(height > alpha * log2(size)).

Another issue was that the original buildIndex had a bug: it called
index.put(gram, index.get(gram).add(p)) but ArrayList.add() returns a boolean,
not the list. We fixed this by calling add() and then put() separately.
